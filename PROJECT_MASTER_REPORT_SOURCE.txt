1. EXECUTIVE SUMMARY & RESEARCH MOTIVATION
================================================================================
Traditional Software-Defined Networking (SDN) decouples the control plane (e.g., 
ONOS, OpenDaylight) from the forwarding data plane (OpenFlow switches). While this 
separation enables centralized programming of forwarding tables, manual generation 
and deployment of low-level OpenFlow flow rules remain a primary source of network 
vulnerabilities, rule-space collisions, and static configuration loops. Intent-Based 
Networking (IBN) attempts to bridge this gap by enabling administrators to express 
operational goals in high-level terms (e.g., "Allow HTTP from Host A to Host B").

Legacy IBN systems, however, rely on static regular expressions (Regex) or strict 
Context-Free Grammars (CFGs). When operators submit colloquial phrasing, non-standard 
abbreviations, or dynamic multi-stage commands, these translation systems fail. 
Large Language Models (LLMs) can solve this grammatical rigidity, but querying 
external commercial cloud APIs (like OpenAI's GPT-4) exposes severe limitations:
1. Data Privacy Violations: Sending network topologies, active IP lists, subnets, and 
   security policies to external hosts violates corporate compliance and data sovereignty.
2. Unpredictable Latency: Wide-area network (WAN) internet jitter makes cloud APIs 
   unstable for time-sensitive network remediation.
3. Lack of Context: LLMs will hallucinate non-existent switch ports or hosts unless 
   grounded with live topological state maps.

LLM-NetAuto-SDN resolves these issues by implementing a completely local, closed-loop 
network automation and self-healing framework. Running an edge-native, quantized 
Llama-3-8B model locally via Ollama ensures complete data sovereignty and sub-second 
inference speeds. To prevent hallucinations, the model is grounded using a local 
ChromaDB vector database via Retrieval-Augmented Generation (RAG). Simultaneously, 
proactive data plane protection is achieved by coupling this generative intent parser 
with an unsupervised online Isolation Forest anomaly detector. High false-alarm rates 
under benign traffic bursts (common in unsupervised machine learning) are eliminated 
via a deterministic, multi-conditional Signature-Based False Positive Filter. The 
system operates a closed feedback loop: anomalies are detected, validated, converted 
into natural language remediation instructions, compiled into OpenFlow rules, and 
deployed programmatically in under 2 seconds.

2. FIVE-PLANE SYSTEM ARCHITECTURE
================================================================================
The architecture is structured across five distinct operational planes, forming a 
continuous closed-loop monitoring and self-healing control flow:

--------------------------------------------------------------------------------
A. THE SEMANTIC INTENT COMPILATION PLANE (PLANE 1)
--------------------------------------------------------------------------------
This plane manages the entry point for human operator commands and autonomous 
remediation actions. It runs a localized LangChain Expression Language (LCEL) chain 
connected to Ollama.
* Core LLM Engine: Llama-3-8B (Quantized Q8_0 profile).
* Syntax Verification: Output formatting is enforced using a strict Pydantic schema 
  defining standard OpenFlow parameters (match rules, IP protocols, ports, and action 
  payloads like "output", "drop", or "set_field").
* Deterministic Fallback: Standard housekeeping tasks (e.g., "Clear all flow rules") 
  are matched against a local regex cache. This bypasses LLM compilation and runs in 
  11.2\,ms, protecting emergency operations from resource starvation or timeouts.

--------------------------------------------------------------------------------
B. THE TOPOLOGY-AWARE RAG PLANE (PLANE 2)
--------------------------------------------------------------------------------
To map unstructured intents to valid physical endpoints, this plane maintains 
real-time topology awareness.
* Topology Seeding: Live switches ($\mathcal{S}$), active hosts ($\mathcal{H}$), and 
  link matrices ($\mathcal{E}$) are serialized into metadata documents.
* Embedding Generation: Generated using the `all-MiniLM-L6-v2` transformer model.
* Vector Database: ChromaDB persistent vector store.
* Context Grounding: During intent parsing, ChromaDB retrieves the top $k=3$ nearest 
  semantic exemplars, subnets, and active node identifiers. This context is injected 
  directly into the LLM system prompt, eliminating hallucinations.

--------------------------------------------------------------------------------
C. THE ASYNCHRONOUS TELEMETRY HARVESTING PLANE (PLANE 3)
--------------------------------------------------------------------------------
Non-intrusive traffic monitoring is handled by a background polling daemon.
* Polling Interval: $\Delta t = 5$s.
* API Northbound Interface: Polls ONOS device port counters or the simulation register.
* Feature Capture: Harvests cumulative counter variables:
  - `BytesReceived`
  - `BytesTransmitted`
  - `PacketsReceived`
  - `PacketsRxDropped`
  - `PacketsRxErrors`
* Feature Engineering: Computes fractional rate metrics between successive polling 
  cycles to construct a 5-dimensional rate feature vector $\mathbf{X}_i$.

--------------------------------------------------------------------------------
D. THE MACHINE LEARNING ANOMALY DETECTION & FILTERING PLANE (PLANE 4)
--------------------------------------------------------------------------------
This plane provides online outlier detection and false-alarm suppression.
* Unsupervised Classifier: Scikit-learn Isolation Forest ensemble ($T = 100$ trees).
* Anomaly Threshold: $\tau = 0.60$.
* False-Alarm Suppression: Outliers are validated against a Signature-Based Filter.
* Threat Classification: Maps verified anomalies to five specific network threats:
  - `"packet_drop"`: Port packet drops exceed 5%.
  - `"error_spike"`: Port error rates exceed 1%.
  - `"traffic_spike"`: General port throughput exceeds 50\,MB/s.
  - `"ddos_sim"`: Volumetric asymmetric packet flooding (RX rate > 10x TX rate).
  - `"bandwidth_hog"`: Persistent high transmit rate (>20\,MB/s).
* Suppression Logic: Unmatched outliers are suppressed as benign traffic bursts.

--------------------------------------------------------------------------------
E. THE CLOSED-LOOP SELF-HEALING PLANE (PLANE 5)
--------------------------------------------------------------------------------
The remediation plane bridges anomaly validation and programmatic self-healing.
* Remediation Dispatcher: Triggered immediately when the signature filter confirms a threat.
* Command Synthesis: Dynamically generates a natural language remediation command:
  "Block all traffic from malicious host h3 (10.0.0.6) to victim host h1 (10.0.0.1) on switch s3"
* Intent Execution: Commands are routed back to the Semantic Intent Parser. The 
  compiled OpenFlow drop rule JSON is pushed to the ONOS controller's REST endpoint 
  and installed on the switches, immediately draining congested switch queues.

3. MATHEMATICAL METHODOLOGY & FORMULATIONS
================================================================================

--------------------------------------------------------------------------------
A. TELEMETRY RATE CONVERSIONS
--------------------------------------------------------------------------------
To compute real-time rate dynamics, raw cumulative counters are processed over a 
polling interval $\Delta t = t - (t - \Delta t)$:

1. Inbound Throughput Rate (MB/s):
   Throughput_rx (t) = [BytesReceived(t) - BytesReceived(t - dt)] / (dt * 10^6)

2. Outbound Throughput Rate (MB/s):
   Throughput_tx (t) = [BytesTransmitted(t) - BytesTransmitted(t - dt)] / (dt * 10^6)

3. Inbound Packet Rate (pps):
   PacketRate_rx (t) = [PacketsReceived(t) - PacketsReceived(t - dt)] / dt

4. Port Packet Drop Rate (fraction):
   DropRate (t) = [PacketsRxDropped(t) - PacketsRxDropped(t - dt)] / 
                  max(1, PacketsReceived(t) - PacketsReceived(t - dt))

5. Port Packet Error Rate (fraction):
   ErrorRate (t) = [PacketsRxErrors(t) - PacketsRxErrors(t - dt)] / 
                   max(1, PacketsReceived(t) - PacketsReceived(t - dt))

* The denominator in (4) and (5) is bounded via a max(1, ...) operation to prevent 
  division-by-zero errors when the port is idle.
* These rate calculations construct the feature vector:
  X_i = [Throughput_rx, Throughput_tx, PacketRate_rx, DropRate, ErrorRate]

--------------------------------------------------------------------------------
B. ISOLATION FOREST MATHEMATICS
--------------------------------------------------------------------------------
The anomaly score of a rate vector $x$ in a dataset of size $n$ is calibrated 
against the expected path length of a Binary Search Tree (BST):

Average path length calibration:
c(n) = 2 * ln(n - 1) + 0.5772156649 - 2*(n - 1)/n

Where 0.5772156649 is Euler's constant. The normalized anomaly score $s(x, n)$ 
across an ensemble of $T = 100$ isolation trees is:

s(x, n) = 2 ^ (- E(h(x)) / c(n))

Where E(h(x)) represents the expected path length of sample $x$ across the forest:
* If s -> 1: E(h(x)) -> 0, sample isolates near the roots and is flagged as an outlier.
* If s -> 0: E(h(x)) -> n-1, sample lies deep in the trees and is marked as normal.
* If s -> 0.5: Sample sits at the average BST path depth, indicating a neutral state.
* Classification Rule: Flag outlier if s(X_i, n) > 0.60.

--------------------------------------------------------------------------------
C. QUEUE DRAINAGE & REMEDIATION CONVERGENCE
--------------------------------------------------------------------------------
The switch buffer queue occupancy $Q_s(t)$ at port $s$ transitions according to the 
incoming traffic rate $\lambda_s(t)$ and the switch service rate $\mu_s(t)$:

Q_s(t + delta) = max(0, Q_s(t) + \int_{t}^{t+delta} (lambda_s(tau) - mu_s(tau)) dtau)

During volumetric attack injections, the arrival rate spikes such that $\lambda_s(t) > \mu_s(t)$, 
causing the buffer to saturate and trigger drops ($f_4 > 0.05$). When our self-healing 
engine deploys OpenFlow block rules at remediation epoch $t_{\text{remed}}$, the arrival 
rate falls immediately via the Heaviside step function:

lambda_s(t) = lambda_s_normal(t) + lambda_s_attack(t) * H(t_remed - t)

The queue drainage time $T_{\text{drain}}$ post-remediation is modeled as:

T_drain = Q_s(t_remed) / (mu_s(t) - lambda_s_normal(t))

This mathematical model ensures swift buffer clearance and latency normalization 
post-remediation.

4. DETAILED FILE-BY-FILE CODEBASE BLUEPRINT
================================================================================
The project is built using a highly modular python workspace. Below is an exhaustive 
blueprint detailing each directory and script:

--------------------------------------------------------------------------------
A. THE GENERATIVE PARSING DIRECTORY (`llm/`)
--------------------------------------------------------------------------------
1. `llm/intent_parser.py`:
   - Enforces the Pydantic JSON schema (`FlowRuleSchema`) defining OpenFlow 
     match criteria (ingress port, source/destination IP, IP protocol, TCP/UDP 
     source/destination ports) and action lists (drop, output, set_field).
   - Establishes the LangChain Expression Language (LCEL) chain connecting the 
     local Ollama Llama-3 client with prompt templates.
   - Implements the `Deterministic Fallback Parser`: maps standard administrative 
     tasks (e.g., "Clear all flow rules", "Remove blockages") to a regex dictionary, 
     bypassing LLM inference in 11.2\,ms to secure emergency operations.
2. `llm/rag_engine.py`:
   - Extracts switch tables, active hosts, and link mappings via ONOS REST calls.
   - Serializes the topology state into standard text documents.
   - Uses `sentence-transformers` (`all-MiniLM-L6-v2`) to generate vector representations.
   - Indexes embeddings in a persistent ChromaDB database.
   - Queries ChromaDB using cosine similarity to retrieve similar past intents (few-shot 
     exemplars) and local node labels ($k=3$), injecting them into the LLM system prompt.
3. `llm/agent.py`:
   - Wraps the LangChain client with logical tool-routing agents, enabling multi-stage 
     operator planning and interactive troubleshooting.

--------------------------------------------------------------------------------
B. THE PROGRAMMATIC FLOW COMPILERS (`netconfig/`)
--------------------------------------------------------------------------------
1. `netconfig/intent_builder.py`:
   - Compiles high-level logical instructions into ONOS Intent structures.
   - Formats JSON requests for the ONOS `intents` REST endpoint, managing host-to-host 
     or point-to-point intents.
2. `netconfig/flow_builder.py`:
   - Compiles low-level device configurations into OpenFlow 1.3 flow rule matches 
     and actions.
   - Formats JSON payloads for the ONOS `flows` REST endpoint, managing match tables, 
     priorities, timeouts, and action instructions.

--------------------------------------------------------------------------------
C. THE SDN CONTROLLER DRIVER (`controller/`)
--------------------------------------------------------------------------------
1. `controller/onos_client.py`:
   - Exposes REST calls for ONOS northbound endpoints:
     - `get_devices()`: Returns active switch lists.
     - `get_links()`: Returns switch interconnects.
     - `get_hosts()`: Returns host MAC/IP and switch interface bindings.
     - `get_flows()`: Returns active forwarding rules.
     - `post_flow_rule(switch_id, flow_json)`: Installs an OpenFlow forwarding rule.
     - `delete_flow_rule(switch_id, flow_id)`: Removes an active flow rule.

--------------------------------------------------------------------------------
D. THE ANOMALY AND SELF-HEALING DRIVER (`monitoring/`)
--------------------------------------------------------------------------------
1. `monitoring/telemetry_collector.py`:
   - Spawns a background thread polling switch ports via ONOS port stats APIs 
     at interval $\Delta t = 5$s.
   - Records cumulative port counters (`bytesRx`, `bytesTx`, `packetsRx`, `packetsRxDropped`, 
     `packetsRxErrors`).
   - Computes fractional rate deltas over successive polling cycles to construct the 
     5-dimensional rate feature vector $\mathbf{X}_i$.
2. `monitoring/anomaly_detector.py`:
   - Runs an online Isolation Forest model (scikit-learn ensemble of 100 trees).
   - Flag sample as outlier if normalized anomaly score $s(x, n) > 0.60$.
   - Runs the deterministic Signature-Based Filter to map outliers to known threat profiles 
     (`packet_drop`, `error_spike`, `traffic_spike`, `ddos_sim`, `bandwidth_hog`), suppressing 
     unmatched anomalies.
3. `monitoring/alert_manager.py`:
   - Tracks anomaly history, aggregates port-level alerts, and manages log registers.
4. `monitoring/feedback_loop.py`:
   - Automates the self-healing pipeline.
   - Intercepts verified threat profiles, generates natural language remediation instructions 
     (e.g., "Block h3 on switch s3"), queries the intent parser, and deploys the OpenFlow 
     blocking rule to ONOS to clear congestion in 1.9 seconds.

--------------------------------------------------------------------------------
E. THE HIGH-FIDELITY NETWORK GRAPH SIMULATOR (`simulation/`)
--------------------------------------------------------------------------------
1. `simulation/network_sim.py`:
   - Emulates northbound ONOS REST endpoints (devices, hosts, links, flows) when 
     operating in offline/simulation mode.
2. `simulation/telemetry_sim.py`:
   - Generates simulated switch port stats, emulating normal Gaussian traffic 
     fluctuations, periodic high-throughput bursts, and specific attack profiles (DDoS, 
     packet drops, error spikes).
3. `simulation/llm_sim.py`:
   - Simulates Ollama schema compilation outputs in offline mode, allowing robust system 
     validation without local GPU hardware.

--------------------------------------------------------------------------------
F. THE FRONTEND Streamlit INTERFACE (`dashboard/`)
--------------------------------------------------------------------------------
1. `dashboard/app.py`:
   - Entry point for the Streamlit dashboard.
   - Coordinates state registers, active routes, and background thread lifecycles.
2. `dashboard/api.py`:
   - FastAPI server exposing 19 REST endpoints for telemetry monitoring, ChromaDB seeding, 
     Ollama configuration, path tracing, and real-time alert updates.
3. `dashboard/pages/`:
   - Modular dashboard pages:
     - `1_Health.py`: Visualizes active alerts, anomaly scores, and self-healing logs.
     - `2_Topology.py`: Renders the live Vis.js network topology.
     - `3_Flows.py`: Displays active flow rules.
     - `4_Realtime.py`: Plots live telemetry throughput, drop, and error rates.
     - `5_RAG.py`: Manages ChromaDB vector seeding and query evaluations.
     - `6_ML_History.py`: Audits historical Isolation Forest outlier logs.
     - `7_Benchmark.py`: Compares Ollama quantization latency and throughput.

5. INFRASTRUCTURE SETUP, SIMULATION, AND RUNTIME DEPLOYMENT
================================================================================
Follow these steps to deploy and run LLM-NetAuto-SDN:

--------------------------------------------------------------------------------
A. PRE-REQUISITES & ENVIRONMENT SETUP
--------------------------------------------------------------------------------
1. Install system dependencies:
   $ sudo apt-get update
   $ sudo apt-get install -y mininet docker.io docker-compose openvswitch-switch
2. Start OpenvSwitch:
   $ sudo service openvswitch-switch start
3. Install Python dependencies:
   $ python -m venv venv
   $ source venv/bin/activate
   $ pip install -r requirements.txt
4. Configure local Ollama server:
   - Download Ollama: https://ollama.com
   - Start Ollama service:
     $ ollama serve
   - Pull the Llama-3-8B model:
     $ ollama pull llama3:8b

--------------------------------------------------------------------------------
B. PRODUCTION DEPLOYMENT VIA MININET & ONOS
--------------------------------------------------------------------------------
To run the full production pipeline with live network emulation:
1. Initialize the docker-compose stack to start the ONOS controller:
   $ docker-compose up -d
   (ONOS will listen on REST port 8181 and OpenFlow port 6653)
2. Deploy the OpenFlow 1.3 Mininet topology (3 switches, 6 hosts):
   $ sudo python production_topology.py
   (This creates hosts h1-h6 and switches s1-s3, connecting them to ONOS)
3. Launch the FastAPI backend and Streamlit dashboard:
   $ ./deploy_production.sh
   (The dashboard opens at http://localhost:8501, and the API runs at http://localhost:8000)

--------------------------------------------------------------------------------
C. HIGH-FIDELITY SIMULATION MODE (OFFLINE MODE)
--------------------------------------------------------------------------------
To run, evaluate, and test the full closed-loop framework on a standard laptop 
without Docker, Mininet, or local GPU hardware:
1. Start the FastAPI backend and Streamlit interface in simulation mode:
   $ ./run.sh
2. Open http://localhost:8501 in a browser.
3. The dashboard and background telemetry collector will run using high-fidelity 
   simulators (`simulation/network_sim.py`, `simulation/telemetry_sim.py`, and 
   `simulation/llm_sim.py`).
4. This mode emulates real port stats, runs the active Isolation Forest classifier, 
   simulates RAG vector queries, and demonstrates the 1.9-second self-healing loop.

7. STEP-BY-STEP CLOSED-LOOP SELF-HEALING SCENARIOS (TRACE ANALYSIS)
================================================================================
Below is a step-by-step trace of the closed-loop system defending a 3-switch, 6-host 
Mininet topology during a simulated volumetric DDoS attack:

1. [t = 0.0s] Telemetry Harvester:
   The background telemetry collector polls switch $s3$ (host $h3$ flooding host $h1$ 
   with UDP packets). It registers a massive increase in inbound port counters:
   - Inbound packet rate: 215,000 pps ($f_3 > 10,000$).
   - Inbound throughput rate: 104 MB/s ($f_1 > 50$).
   - Drop rate: 0.08 ($f_4 > 0.05$).

2. [t = 0.1s] Anomaly Detection:
   - The Isolation Forest flags the telemetry vector as an outlier, calculating a 
     normalized anomaly score of $s(x, n) = 0.85$.
   - The vector is sent to the Signature Filter. The filter evaluates:
     `f_1 > 10 * f_2` (104 MB/s RX vs 2 MB/s TX) and `f_3 > 10,000 pps`.
   - The signature matches, confirming the threat as `"ddos_sim"`.

3. [t = 0.3s] Closed-Loop Remediation:
   - The Alert Manager triggers the Feedback Loop.
   - The engine queries ChromaDB, retrieves the physical topology state (identifying 
     h3 on switch s3, ingress port 3), and generates a natural language mitigation 
     command:
     "Block all traffic from malicious host h3 (10.0.0.6) to victim host h1 (10.0.0.1) on switch s3"

4. [t = 1.8s] Semantic Parsing:
   - The localized Llama-3 model processes the command, matches the target switch s3 
     and ingress port, and compiles the Pydantic-verified JSON OpenFlow rule:
     ```json
     {
       "switch_id": "of:0000000000000003",
       "priority": 50000,
       "timeout": 60,
       "match": {
         "in_port": 3,
         "eth_type": "0x0800",
         "ipv4_src": "10.0.0.6",
         "ipv4_dst": "10.0.0.1"
       },
       "actions": [
         { "type": "drop" }
       ]
     }
     ```

5. [t = 1.9s] Data Plane Enforcement:
   - The FastAPI backend pushes this OpenFlow drop rule to the ONOS controller's REST 
     endpoint.
   - Switch s3 immediately installs the flow rule.
   - Packets from h3 are dropped at the ingress port. Switch queues drain, packet drops 
     return to 0, and normal communication is restored.
